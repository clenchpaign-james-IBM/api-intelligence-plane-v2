# Enhanced Pattern Simulator Guide

## Overview

The Enhanced Pattern Simulator (`enhanced_pattern_simulator.py`) is designed specifically for webmethods API Gateway to generate realistic traffic patterns that produce actual transaction logs and events.

## Key Design Principles

### 1. Real Gateway Invocations
- **All requests must reach the gateway** to generate transaction logs
- No reliance on query parameters that the gateway doesn't understand
- Patterns are simulated through actual HTTP behavior (timing, volume, endpoints)

### 2. Pattern Implementation Strategy

Since webmethods API Gateway doesn't understand custom query parameters, patterns are simulated through:

- **Request Timing**: Varying intervals between requests (bursts, ramps, steady)
- **Endpoint Variations**: Using invalid paths/IDs to generate real 404/400 errors
- **Authentication Variations**: Alternating valid/invalid credentials for real 401/403 errors
- **Volume Control**: Adjusting RPS dynamically to trigger rate limits
- **Shadow Endpoints**: Probing undocumented paths to simulate security threats

### 3. Error Generation

Instead of sending query parameters like `?error=true`, we generate real errors by:
- Appending invalid resource IDs: `/api/resource/999999`
- Using non-existent paths: `/api/nonexistent_endpoint`
- Sending invalid credentials: `HTTPBasicAuth("invalid", "invalid")`
- Overwhelming rate limits: Progressively increasing RPS

## Supported Patterns

### 1. Steady (`steady`)
**Purpose**: Baseline traffic for normal operations

**Implementation**:
- Constant RPS throughout duration
- All requests to valid endpoint
- Minimal error injection

**Use Case**: Establishing baseline metrics

**Example**:
```bash
python backend/scripts/enhanced_pattern_simulator.py \
  --url "http://localhost:5555/gateway/PetStore/1.0.0/pet/1" \
  --pattern steady \
  --rps 5 \
  --duration 60
```

### 2. Increasing Error Trend (`increasing-error-trend`)
**Purpose**: Simulate gradual service degradation

**Implementation**:
- Error probability increases linearly from 5% to 85%
- Errors generated by requesting invalid endpoints
- Progress-based error injection: `error_rate = 0.05 + (progress * 0.75)`

**Use Case**: Testing prediction models for service degradation

**Example**:
```bash
python backend/scripts/enhanced_pattern_simulator.py \
  --url "http://localhost:5555/gateway/PetStore/1.0.0/pet/1" \
  --pattern increasing-error-trend \
  --rps 5 \
  --duration 120
```

**Expected Behavior**:
- First 20%: ~5-20% errors
- Middle 40%: ~30-50% errors  
- Last 40%: ~60-85% errors

### 3. Bucketed Error Trend (`bucketed-error-trend`)
**Purpose**: Simulate step-wise service degradation

**Implementation**:
- Divides requests into 6 buckets
- Each bucket has progressively higher error rate
- Bucket error rates: 5%, 20%, 35%, 50%, 65%, 80%

**Use Case**: Testing time-bucketed metrics aggregation

**Example**:
```bash
python backend/scripts/enhanced_pattern_simulator.py \
  --url "http://localhost:5555/gateway/PetStore/1.0.0/pet/1" \
  --pattern bucketed-error-trend \
  --rps 10 \
  --duration 60
```

**Expected Behavior**:
- Clear step-wise increases in error rate
- Each bucket represents ~10 seconds of traffic
- Ideal for validating metrics bucketing logic

### 4. Burst (`burst`)
**Purpose**: Simulate periodic traffic spikes

**Implementation**:
- 12-second cycles: 3s burst + 9s quiet
- During burst: 5x RPS with 30% error rate
- During quiet: Normal RPS with minimal errors

**Use Case**: Testing burst handling and capacity planning

**Example**:
```bash
python backend/scripts/enhanced_pattern_simulator.py \
  --url "http://localhost:5555/gateway/PetStore/1.0.0/pet/1" \
  --pattern burst \
  --rps 5 \
  --duration 60
```

**Expected Behavior**:
- Periodic spikes in request rate
- Increased errors during burst windows
- ~5 complete cycles in 60 seconds

### 5. Traffic Ramp-Up (`traffic-ramp-up`)
**Purpose**: Simulate gradual load increase

**Implementation**:
- Starts at 10% of base RPS
- Ends at 200% of base RPS
- Linear increase over duration

**Use Case**: Testing auto-scaling and capacity planning

**Example**:
```bash
python backend/scripts/enhanced_pattern_simulator.py \
  --url "http://localhost:5555/gateway/PetStore/1.0.0/pet/1" \
  --pattern traffic-ramp-up \
  --rps 10 \
  --duration 120
```

**Expected Behavior**:
- Gradual increase from 1 RPS to 20 RPS
- Smooth transition without sudden spikes
- Tests gateway's ability to handle increasing load

### 6. Error Burst (`error-burst`)
**Purpose**: Simulate concentrated error windows

**Implementation**:
- 20-request cycles
- Requests 9-13 (5 requests) generate errors
- Other requests are normal

**Use Case**: Testing error detection and alerting

**Example**:
```bash
python backend/scripts/enhanced_pattern_simulator.py \
  --url "http://localhost:5555/gateway/PetStore/1.0.0/pet/1" \
  --pattern error-burst \
  --rps 5 \
  --duration 60
```

**Expected Behavior**:
- Periodic windows of concentrated errors
- Clear error spikes in metrics
- ~15 error bursts in 60 seconds

### 7. Rate Limit Hit (`rate-limit-hit`)
**Purpose**: Trigger gateway rate limiting

**Implementation**:
- Progressively increases RPS
- Speed multiplier: `min(20, 3 + request_number // 10)`
- Starts at 3x base RPS, increases to 20x

**Use Case**: Testing rate limiting policies and 429 responses

**Example**:
```bash
python backend/scripts/enhanced_pattern_simulator.py \
  --url "http://localhost:5555/gateway/PetStore/1.0.0/pet/1" \
  --pattern rate-limit-hit \
  --rps 5 \
  --duration 30
```

**Expected Behavior**:
- Rapid increase in request rate
- 429 (Too Many Requests) responses
- Rate limit policy activation

### 8. Auth Failures (`auth-failures`)
**Purpose**: Simulate authentication issues

**Implementation**:
- Alternates between valid and invalid credentials
- Even requests: `HTTPBasicAuth("invalid_user", "invalid_password")`
- Odd requests: Original credentials (or None)

**Use Case**: Testing authentication monitoring and security alerts

**Example**:
```bash
python backend/scripts/enhanced_pattern_simulator.py \
  --url "http://localhost:5555/gateway/PetStore/1.0.0/pet/1" \
  --pattern auth-failures \
  --rps 5 \
  --duration 60 \
  --username "valid_user" \
  --password "valid_pass"
```

**Expected Behavior**:
- 50% of requests return 401/403
- Clear authentication failure pattern
- Tests security monitoring

### 9. Shadow API Probe (`shadow-api-probe`)
**Purpose**: Simulate security reconnaissance

**Implementation**:
- Every 3rd request probes shadow endpoints
- Shadow paths include:
  - `/api/internal/debug`
  - `/api/v1/admin/users`
  - `/api/legacy/v0/data`
  - `/actuator/health`
  - `/.well-known/security.txt`

**Use Case**: Testing shadow API detection and security monitoring

**Example**:
```bash
python backend/scripts/enhanced_pattern_simulator.py \
  --url "http://localhost:5555/gateway/PetStore/1.0.0/pet/1" \
  --pattern shadow-api-probe \
  --rps 5 \
  --duration 60
```

**Expected Behavior**:
- ~33% requests to undocumented endpoints
- Mix of 404 and potentially 403 responses
- Triggers shadow API detection algorithms

### 10. Response Time Patterns

**Decreasing Response Time** (`decreasing-response-time`):
- Relies on actual gateway response times
- Uses steady traffic pattern
- Gateway must naturally improve response times

**Increasing Response Time** (`increasing-response-time`):
- Relies on actual gateway response times
- Uses steady traffic pattern
- Gateway must naturally degrade response times

**Decreasing Availability** (`decreasing-availability`):
- Similar to increasing-error-trend
- Simulates progressive service unavailability

## Integration with Intelligence Pipeline

### 1. Transaction Log Generation
All requests generate real transaction logs in webmethods API Gateway, which are then:
- Collected by `transactional_logs_collection_jobs.py`
- Stored in OpenSearch `gateway-transactions-*` indices
- Analyzed by intelligence pipeline

### 2. Metrics Aggregation
Patterns generate time-series data that feeds into:
- Time-bucketed metrics storage
- Prediction models
- Optimization recommendations
- Security analysis

### 3. Pattern Detection
The intelligence pipeline can detect:
- Error trends (increasing/bucketed)
- Traffic anomalies (bursts/ramps)
- Security threats (shadow API probes)
- Performance degradation (response times)

## Best Practices

### 1. Start Small
Begin with low RPS and short duration to verify connectivity:
```bash
python backend/scripts/enhanced_pattern_simulator.py \
  --url "http://localhost:5555/gateway/PetStore/1.0.0/pet/1" \
  --pattern steady \
  --rps 1 \
  --duration 10
```

### 2. Monitor Gateway
Watch gateway logs and metrics during simulation:
- Check transaction logs in webmethods
- Monitor OpenSearch indices
- Verify data collection jobs are running

### 3. Gradual Load Increase
Don't jump directly to high RPS:
```bash
# Start with 5 RPS
--rps 5 --duration 60

# Then increase to 10 RPS
--rps 10 --duration 60

# Finally test high load
--rps 50 --duration 30
```

### 4. Pattern Sequencing
Run patterns in logical sequence for comprehensive testing:
```bash
# 1. Establish baseline
--pattern steady --rps 5 --duration 60

# 2. Test error handling
--pattern increasing-error-trend --rps 5 --duration 120

# 3. Test burst handling
--pattern burst --rps 5 --duration 60

# 4. Test rate limiting
--pattern rate-limit-hit --rps 10 --duration 30

# 5. Test security
--pattern shadow-api-probe --rps 5 --duration 60
```

## Troubleshooting

### Connection Refused
**Problem**: Cannot connect to gateway
**Solution**: Verify gateway is running and URL is correct
```bash
curl -v http://localhost:5555/gateway/PetStore/1.0.0/pet/1
```

### No Transaction Logs
**Problem**: Requests succeed but no logs in OpenSearch
**Solution**: 
1. Check gateway transaction logging is enabled
2. Verify data collection jobs are running
3. Check OpenSearch connectivity

### Rate Limiting Not Triggered
**Problem**: `rate-limit-hit` pattern doesn't trigger 429 responses
**Solution**:
1. Verify rate limit policy is configured on the API
2. Increase base RPS: `--rps 20`
3. Reduce duration to concentrate load: `--duration 15`

### High Error Rate
**Problem**: Too many errors even in steady pattern
**Solution**:
1. Verify API endpoint is valid
2. Check authentication credentials
3. Test endpoint manually first

## Comparison with Original Script

### Original (`live_api_invoker_rps.py`)
- Sends query parameters to simulate patterns
- Gateway ignores these parameters
- Limited real error generation
- Simpler implementation

### Enhanced (`enhanced_pattern_simulator.py`)
- Generates real traffic patterns through actual behavior
- All requests reach gateway and generate logs
- Comprehensive error injection strategies
- Object-oriented design with pattern classes
- Better result tracking and statistics
- More realistic simulation
